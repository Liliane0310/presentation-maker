Skill 是什么？
—— 让 AI 真正“会做事”的那个能力块

一份手绘纸感的入门讲解：从 Prompt、Skill 到 Workflow 的位置关系，帮新手先把地图搭起来。

“Prompt 像一句请求，Skill 像一个专长，Workflow 像把这些专长串起来的工作线。”
这一页先把情绪放稳。很多人刚接触 AI 工具的时候，会听到 Prompt、Skill、Workflow、Agent 一串词，感觉每个都懂一点，但连起来就乱了。

今天我们先不追求一口气搞懂全部，而是只抓住一个核心词：Skill。如果用一句话概括，Skill 就像 AI 手边那个“会干某件事的人”。它不是空泛能力，而是一个能稳定完成动作的模块。

接下来我会用 6 页，把它讲成一个新手也能立刻记住、立刻拿去判断产品设计的概念。

01 · 什么是 Skill

Skill（技能）= 一个可被调用、可重复使用、输入输出相对明确的能力模块。它回答的是：“我会做什么？”

输入 Input
→
执行能力
→
输出 Output
一句话：Skill 不是一句聪明的话，而是一个能被反复调用的“专长单元”。
这里先给定义。Skill 不是泛泛地说“我很会写东西”，而是把能力压缩成一个可调用模块。它要能回答三个问题：输入是什么、执行什么动作、输出什么结果。

比如“翻译一段文字”“搜索一个关键词”“把口语改成正式文案”，这些都更像 Skill。因为它们的边界相对清楚，别人也能反复调用。

所以 Skill 的重点不在于听起来厉不厉害，而在于它是不是稳定、清楚、可复用。这个感觉先记住，后面就不容易和 Workflow 混了。

02 · 一个好 Skill 的 3 个特征

🎯 边界清楚

知道自己负责什么，不把很多不同逻辑硬塞进一个能力里。

🔁 可以复用

同一个 Skill 能被多个场景调用，而不是只服务一条临时链路。

📦 输入输出明确

别人知道要给它什么，也知道会拿回什么，便于串接到更大流程里。

什么样的 Skill 才算好？我建议先看三个特征。第一，边界要清楚。一个 Skill 最怕什么都想做，最后变成一个说不清也测不准的黑箱。

第二，要能复用。今天帮你写标题，明天帮你整理摘要，后天还能被别的流程再次调用，这才叫能力资产，而不是一次性脚本。

第三，输入输出要明确。别人知道该喂给它什么，也知道会拿回什么。只要这三点站住了，Skill 才有机会被更大的流程真正用起来。

03 · Skill 和 Prompt 不一样

维度PromptSkill
定位一句请求或指令一个可调用能力
稳定性依赖上下文和表述方式更强调固定输入输出与执行边界
复用性可以复制，但不一定可组合天然适合被多个流程反复调用
类比一句对话一个工种
Prompt 更像“你怎么说”，Skill 更像“它会什么”。
很多人第一次接触 Skill，会下意识把它理解成“高级 Prompt”。这不完全对。Prompt 更像一句请求，重点在你怎么说；Skill 更像一个能力模块，重点在它会什么、能稳定做到什么程度。

你当然可以用 Prompt 驱动一个 Skill，但两者不是一层东西。一个 Prompt 可以临时很聪明，但未必可复用；一个 Skill 就算底层有 Prompt，也更像被包装好的工种。

所以判断一个东西是不是 Skill，不要只看它写了多少话，而要看它是不是已经长成一个别人能调用的能力单元。

04 · Skill 在系统里通常做什么

🔎 搜索类 Skill

查文档、查网页、查库，负责把外部信息找回来。

✍️ 生成类 Skill

写摘要、写标题、写邮件、写提纲，把内容产出来。

🧹 处理类 Skill

清洗文本、抽取字段、分类、改写，让原始内容变得可用。

📤 执行类 Skill

发消息、改文件、提交工单、调用 API，把动作真正落出去。

这一页是帮大家建立手感。Skill 在系统里常见的职责，大概就这几类。第一类是搜索，负责把外部信息找回来；第二类是生成，负责把内容写出来；第三类是处理，负责清洗、提取、分类；第四类是执行，负责真正把动作发出去。

你会发现，Skill 不是一个抽象标签，而是很具体的能力工位。一个系统里往往不是只有一个 Skill，而是很多 Skill 各司其职。等这些能力块准备好了，Workflow 才有东西可以编排。

05 · Skill 和 Workflow 的关系

如果 Skill 是一个个可调用的工种，那么 Workflow 就是把这些工种按顺序组织起来的施工图。

Skill：搜索
→
Skill：提炼
→
Skill：写作
→
Skill：发送
一句话记忆：Skill 负责“会”，Workflow 负责“怎么串起来干”。
讲到这里，就可以把 Skill 放回更大的系统里了。Skill 自己像一个工种，Workflow 像把工种串起来的施工图。你可以只有 Skill，没有 Workflow，这时候每个能力都能单独用，但不一定形成完整链路。

一旦有了 Workflow，它就会决定谁先上场、谁后上场、结果怎么往下传、哪里需要判断、哪里需要兜底。

所以两者不是替代关系，而是上下层关系。Skill 负责“会”，Workflow 负责“怎么把这些会的东西组织成一条真正能跑的线”。

06 · 总结

  • 💡 Skill 是一个可调用、可复用、边界清楚的能力块。
  • 📦 判断它像不像 Skill，要看输入输出是否明确。
  • 📌 Prompt 更像一句请求，Skill 更像一个工种。
  • 🚀 Workflow 不替代 Skill，而是把多个 Skill 编排成可执行链路。
最后一句：先把 Skill 设计清楚，Workflow 才会真正灵活。

最后收个尾。今天这份内容真正想帮你记住的，不是一个术语定义，而是一种判断方式：当你看到一个能力时，问自己三个问题，它负责什么、别人怎么调用它、它会产出什么。

只要这三个问题答得清楚，这个东西就更像 Skill；答不清楚，它可能还只是一个临时想法，或者一段还没工程化的 Prompt。

所以先把 Skill 设计清楚，后面的 Workflow 才有稳定积木可以搭。这个顺序一旦明白，很多产品设计问题都会顺很多。
