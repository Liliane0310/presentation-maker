# presentation-maker

把一个主题整理成可讲、可看、可交付的 HTML 演示稿。默认产物是单个自包含 `.html` 文件，不输出来源列表，不依赖外部 CSS 或 JS 才能完成核心展示。输出样式默认采用“手绘纸感（Paper & Hand-drawn）”16:9 演示文稿风格：画面饱满、标题区清晰、主体内容充分利用版心，像一套精致的纸质手写讲义，而不是网页 landing page 或居中小卡片。

## 默认定位

- 默认演示类型：教学科普型
- 默认听众：初学者、非技术背景或泛业务听众
- 默认 notes：完整逐字稿
- 默认页数：10-20 页
- 默认交付：`主题名.html`
- 默认版式：16:9 横版纸感演示稿
- 默认品牌/署名：如用户未指定，可在封面使用 `.brand-mark` 放置作者或主题标识

## 工作流

### Step 1：先采集三件事

收到需求后，不要直接生成成品。先确认：

1. 听众是谁
   - 完全新手
   - 有基础的从业者
   - 管理层或业务方
   - 通用混合听众
2. notes 粒度
   - 完整逐字稿
   - 关键提示词
   - 不要 notes
3. 必须包含什么
   - 指定案例
   - 指定概念
   - 指定章节
   - 指定工具或术语

如果用户已经给了足够清楚的约束，可以只补最关键的缺口，不必机械重复提问。

### Step 2：先出大纲，等待确认

根据主题、听众和必须内容，先输出页纲，不直接写完整 HTML。

大纲格式建议：

```text
建议大纲（共 N 页）

1. 封面 — 标题 + 副标题 + 总纲句
2. 为什么值得了解这个主题 — 问题或背景
3. 核心概念 — 白话定义 + 类比
4. ...
N. 结语 — 总结 + 下一步
```

大纲规则：

- 每页只承担一个核心观点
- 同一段落内不要连续堆太多术语
- 页面节奏要有变化，避免连续多页都是纯段落
- 用户确认或修改大纲后，才进入生成阶段

### Step 3：生成完整 HTML

用户确认大纲后，再生成完整演示稿：

- 最终交付一个自包含 `.html`
- 页面结构、视觉语言和运行逻辑以 `references/html-template.md` 为唯一骨架来源
- 内容写法遵守 `references/layout-guide.md` 和 `references/script-guide.md`
- 视觉约束遵守 `references/style-tokens.md`
- 不输出来源列表，不附参考文献页
- 首页不写“操作：← / → / Space 翻页”这类观众提示
- 内容少的页面可使用 `.center-content` 垂直居中，内容多的页面保持靠上排版
- 抽象概念优先加入生活化类比页；类比应根据主题临时设计，不固定套用某个具体示例

### Step 4：注入小赖字体（必须）

生成 HTML 后，必须运行 `scripts/embed-font.py` 把小赖字体以 base64 内嵌进 HTML，否则用户打开看不到小赖字体。

```bash
python scripts/embed-font.py <生成的html路径>
```

要求：

- 如果用户机器没有 Python 或缺少 `fonttools`/`brotli`，提示用户安装：`pip install fonttools brotli`
- 如果 `assets/xiaolai.woff2` 不存在，先运行 `python scripts/subset-font.py` 生成（输入字体路径默认 `D:/讯飞实习/xiaolai.ttf`，可改成用户自己的路径）
- 注入是幂等的，重复运行会自动跳过
- 注入会让 HTML 增加约 1.85MB（1.39MB woff2 → base64），这是预期行为
- 注入完成后告知用户：HTML 已自包含小赖字体，可直接双击打开，发给别人也能正常显示

## 输出约束

- 成品必须是单个自包含 `.html`
- 不把 CSS、JS 拆成额外文件交给用户
- 不输出“下面是 HTML 代码，请自行保存”这类教程式说明
- 默认文件名用主题名或主题名拼音
- 第一页必须是强封面页，最后一页必须有收束感
- `notes` 需要和对应 slide 内容一致，不能写成独立散文
- 页面必须充分利用 16:9 画布，不要把正文缩在中央小卡片里
- 英文、数字、UI 文本不要优先使用小赖字体；中文可以回退到小赖字体以保留手绘感
- 导航控件和演讲者模式可以存在，但不要把操作说明写进观众可见正文

## 内容规则

### 解释优先

- 抽象概念优先用生活化类比解释
- 新术语首次出现时，立刻加白话解释
- 默认站在初学者视角组织信息，不炫技，不堆名词
- 如果主题适合，可用静态文字 + 轻量动效类比页来建立直觉；具体比喻必须贴合当前主题

### 页面节奏优先

- 解释页、表格页、卡片页、quote 页要穿插出现
- 避免连续两页都只有大段正文
- 多项并列信息优先拆成卡片或对照结构
- 内容稀疏的页使用 `.center-content`；信息密集、表格、代码、长列表页不要强行居中

### notes 优先口语

- 完整逐字稿默认写成可直接讲的口语
- 不写成论文摘要
- 每页最好有一个开头钩子、一个解释段和一个过渡段

## 修改约束

生成后若用户要求调整，优先定点修改，不默认重写全稿。

支持的常见指令：

- “第 N 页太抽象，换个更生活化的类比”
- “第 N 页加一个案例”
- “加一页讲 xxx”
- “把第 A 页和第 B 页顺序调换”
- “删掉第 N 页”
- “把 notes 改得更口语一些”
- “这一页改成表格”

除非用户明确要求整体重构，否则只改受影响的部分。

## 自检清单

生成完成后，自查：

- [ ] 最终交付物是单个自包含 `.html`
- [ ] 已运行 `scripts/embed-font.py` 注入小赖字体（HTML 体积约 +1.85MB）
- [ ] 每页只有一个核心观点
- [ ] 页面节奏有变化，没有连续多页纯文本
- [ ] 第一页是封面页，最后一页有明确收束
- [ ] 首页没有观众操作提示文字
- [ ] 页面版心饱满，不是居中小卡片
- [ ] 内容少页面已居中，内容多页面没有被强行居中
- [ ] 英文和数字优先使用西文字体，不使用小赖字体
- [ ] 抽象概念优先给了类比
- [ ] 新术语首次出现时有白话解释
- [ ] notes 与当前页内容一致
- [ ] 若为完整逐字稿，语气口语化、可直接讲
- [ ] 演讲者模式、概览和键盘翻页在结构上齐备

## 文件使用说明

- `references/style-tokens.md`
  - 视觉系统和 CSS tokens
- `references/layout-guide.md`
  - 内容场景到页面结构的映射
- `references/script-guide.md`
  - notes 与讲稿写法规范
- `references/html-template.md`
  - 唯一 HTML 骨架模板
- `examples/example-deck.html`
  - 可直接打开的参考成品（已注入小赖字体）
- `scripts/subset-font.py`
  - 子集化脚本：把 `xiaolai.ttf` 转 GB2312 子集 woff2（输出到 `assets/xiaolai.woff2`）
  - 仅当 `assets/xiaolai.woff2` 缺失时才需要运行
- `scripts/embed-font.py`
  - 字体注入脚本：把 `assets/xiaolai.woff2` 以 base64 内嵌进 HTML
  - **每次生成 HTML 后必须运行**
- `assets/xiaolai.woff2`
  - 子集化后的小赖字体（约 1.39MB，覆盖 GB2312 6763 字）

## 执行提醒

- 如果用户只给了主题，不要跳过提问直接生成
- 如果用户没有指定风格，默认保持暖调手绘纸感、16:9 讲义式演示布局
- 如果用户要求专业主题，也要优先保证“讲得懂”，而不是“术语全”
- 生成新的 deck 时，复用模板的结构和节奏，只替换标题、页面内容、页码和 `.notes`
- 默认采用 `references/html-template.md` 的结构语言：
  - 全屏 `deck`
  - 大版心纸张式 slide
  - 底部进度 / 页码 / 翻页控件
  - `notes` 隐藏但可由演讲者模式读取
